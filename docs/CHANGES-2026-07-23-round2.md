## P0 — 부분 동결 × 결정적 check 구조 모순 (round2-B 발견 1) → 해결

- **결함**: `freeze --allow-partial`로 이전 스냅샷에만 pass로 남은 케이스가 생기면, 결정적 `check`가
  그 케이스를 replay하지 못해(최신 단일 스냅샷만 replay-map에 적재) **구조적으로 영구 반려**했고, 그
  error 행이 "assert 실패"로 오분류됐다. README 표본 운영 수칙이 권장하는 정상 루프에서 자연 발생.
- **수정**: replay 소스를 최신 단일 스냅샷 → **floor를 덮는 스냅샷 합집합**으로. 케이스별로 "그 caseId를
  pass:true로 담은 가장 최근 스냅샷의 출력"을 replay한다. probeHash 대조는 케이스별 출처 스냅샷 기준으로
  분리. "replay 불가(스냅샷 커버 안 됨)"를 진짜 assert 실패와 다른 사유로 분류. (floor에서 부분 동결
  케이스를 제외하는 방식은 래칫 철학 위반이라 쓰지 않음.)
- **재테스트 (적용 B)**: 발견 1을 겪은 그 루프를 다시 — 어려운 입력 1건을 `--allow-partial`로 부분
  동결한 상태에서 결정적 `check` → **exit 0**이면 pass(이전엔 영구 반려). 진짜 assert 실패 케이스는
  `[회귀]`, 스냅샷 미커버는 `[replay 불가]`로 **다른 라벨**로 찍히는지 확인.

## P1 — 라이브 케이스별 재시도 (round2-B 발견 2) → `--retry` 신설

- **요청**: 라이브 형식 위반(JSON 앞 서두·코드펜스) 시 해당 케이스만 재생성하는 재시도가 없어, 케이스가
  늘수록 전건 동시 통과가 확률 게임(실측 8회 전패).
- **변경**: `--retry <N>` 신설 (`check --live`·`freeze`, 기본 0=현행). 라이브 eval 후 **실패한 케이스만**
  최대 N회 재평가, 통과 케이스는 재호출 안 함. promptfoo `--filter-pattern`으로 그 케이스만 다시 태운다.
  결정적 replay 경로에선 무의미하므로 무시+경고.
- **중요(개발 중 verifier가 잡은 결함까지 반영)**: 재시도는 정의상 같은 프롬프트로 다시 부르므로 기본
  promptfoo 캐시가 히트해 **provider가 실제로 재실행되지 않는 no-op**이었다. 재시도 호출에
  `PROMPTFOO_CACHE_ENABLED=false`를 강제해 fresh 재생성을 보장하도록 수정했고(초기 eval 캐시는 유지 —
  통과 케이스 확정용), 실 provider 호출 카운트 통합 테스트로 못박았다(수정 전 1회 → 후 2회).
- **재테스트 (적용 B)**: 강화 전 프롬프트(케이스별 형식 위반이 재현되던 그 상태)로
  `check --live --retry 2` → 실패 케이스가 재시도로 통과에 도달하고 verdict에 "caseX: 재시도 k/N에
  통과"가 찍히면 pass. `--retry 0`(또는 미지정)이 이전과 동일하면 하위호환 pass.
- **부산물(발견 2의 프롬프트 데이터)**: 일반 금지문 8전패 → 실제 위반 문구를 negative example로 인용하니
  1회에 7/7. `--retry`는 이 프롬프트 개선을 대체하는 게 아니라 **잔여 비결정성의 안전망**이다.

## lint 적용 경계 (round2 항목 3 FAIL) → 별개 도구로 정직하게 문서화

- **판정**: A에서 `lint`가 브리핑 수제 린터를 대체 못한 건 버그가 아니라 **입력 모델 불일치**다. 계약
  asserts가 vars-gated(원본 대조형)인데 라이브 md엔 vars가 없어 전 프로브 스킵 → false negative(exit 0).
- **변경(문서만, 커맨드 불변)**: README에 lint 적용 조건 명시(계약과 **같은 입력 단위**일 때만 유효,
  vars 의존 프로브는 `--vars` 필수, 없이 부르면 스킵되어 놓침). PROBES.md에 **vars 어댑터 패턴**(앱이
  도메인 문서를 파싱해 (output, vars)를 만들어 주입) + "계약 프로브(회귀 가드)와 라이브 표면 린터는
  별개일 수 있다"는 경계. AGENTS.md에 "vars 없는 lint의 exit 0을 통과로 오인 금지".
- **재테스트 (적용 A)**: 브리핑 md를 파싱해 항목+vars를 만들어 `lint --vars`로 주입하는 어댑터 경로가
  수제 린터와 같은 위반을 잡으면 pass. 그게 과하면 "계약+수제 린터 이중 운영 + SSOT 파생 동기화"
  현행 유지가 정답 — 그 판단을 문서가 지지하는지 확인.
