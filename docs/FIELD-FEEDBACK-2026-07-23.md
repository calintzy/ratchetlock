# 실전 적용 피드백 (2026-07-23)

두 곳에 실운영 적용한 결과를 모은 개발 피드백이다.

- **적용 A (브리핑)**: daily-briefing 쪽 프롬프트에 적용한 세션의 평가.
- **적용 B (물어오리)**: daily-news-reels 뉴스 재작성 프롬프트 계약(`contracts/rewrite/`, 커밋 35e34e6). 실측 기사 6건 라이브 6/6 동결, contracts/** push 시 결정적 check를 도는 CI 게이트(contract-check.yml) 가동 중.

같은 증상이 두 적용에서 독립적으로 재현된 항목은 표시했다.

## 실사용에서 실제로 막힌 것 (P0 — 먼저 고칠 것)

### 1. 빌드·설치·영구 경로 (적용 A에서 발생, 적용 B에서 반증 데이터 있음)

적용 A: npm install의 prepare 훅이 자동 빌드를 못 해서 `npm run build`를 수동으로 돌려야 했고, npm 미배포라 /tmp에 빌드됨 — 재부팅하면 소실. 계약 check를 CI/루틴에 상시로 물리려면 도구가 영구 경로에 안정적으로 있어야 하는데 그 설치법이 없다. npm 배포나 최소한 영구 설치 레시피가 선결 조건.

**적용 B 대조 실측**: GitHub Actions ubuntu-latest에서 `npm install --no-save github:calintzy/ratchetlock`은 prepare 훅 빌드까지 정상 작동했다(contract-check run 1m40s success, run 29930226790). 즉 "prepare가 빌드를 못 한다"는 전역 결함이 아니라 환경 의존 증상일 가능성이 크다 — npm 버전별 git 의존성의 devDependencies(typescript) 설치 정책 차이 의심. 실패 환경의 npm 버전·설치 명령을 재현해 원인을 좁힐 것. 결론(npm 배포 또는 공식 설치 레시피 문서화가 필요하다)은 그대로 유효하다.

### 2. 실전 등록 워크플로 부재 (적용 A·B 공통)

적용 A: demo.sh가 끝에서 ratchet.json을 리셋해버려서, 등록 상태를 남기려고 스크립트를 잘라 재실행해야 했다. "데모 시연"과 "실전 등록"이 한 스크립트에 얽혀 있다.

적용 B 보충: 커맨드를 직접 조합(init → check --live → freeze → check)하면 등록 자체는 되지만, 그 절차가 README에 "실전 등록 퀵스타트"로 정리돼 있지 않아 examples/cardnews를 역산해야 했다. demo를 고치기보다 **REGISTER.md(또는 README 섹션): 새 프롬프트 등록 5단계 + CI 연동 예시(contract-check.yml 실물 참고 가능)**를 추가하는 쪽이 빠르다.

## 장점 극대화 (P1)

### 3. 라이브 lint 모드를 1급 기능으로 (적용 A·B 공통 — 가장 값어치 큰 제안)

ratchetlock은 과거 케이스 회귀 가드만 하는데, 실전 니즈의 절반은 "오늘 새 출력에 같은 프로브 적용"(신규 왜곡 검사)이다.

- 적용 A: 이게 없어서 프로브를 복사한 별도 린터를 손으로 만들어야 했다.
- 적용 B: 계약 asserts.js가 프로덕션 게이트(validate.mjs) 로직의 복사본이 됐고, 둘의 동기화를 지켜주는 게이트가 없다. validate.mjs 규칙을 고치면 asserts.js를 손으로 맞춰야 하며 어긋나면 조용히 벌어진다 — **ratchetlock이 막으려는 바로 그 드리프트를 사용 구조가 스스로 만든다.**

두 적용에서 독립적으로 같은 부채가 생겼으니 우연이 아니라 구조적 결함이다. 해법 두 갈래: ① `ratchetlock lint <output>` — 등록된 프로브를 임의 출력에 적용(과거 재발 + 신규 왜곡을 한 프로브로 커버), ② 프로브를 외부에서 임포트 가능한 단일 모듈로 두는 사용 패턴 지원·가이드(프로덕션 검증기와 계약 프로브가 같은 파일을 require).

### 4. 프로브 드리프트 검출을 전면에 (적용 A)

이게 promptfoo에 없는 진짜 킬러 기능인데 README에 묻혀 있다. "측정기가 조용히 헐거워지는 걸 막는 유일한 도구"라는 포지셔닝을 상단으로.

### 5. status 표기 혼동 (적용 B)

케이스 6건을 동결했는데 `status`는 "frozen: 1건"으로 표시한다(스냅샷 묶음 단위). 처음 보면 동결이 덜 된 걸로 오독한다. "frozen 1건(케이스 6건)"처럼 케이스 수 병기 필요.

## 로드맵 제언 (P2)

### 6. 의미 왜곡 LLM 루브릭 — 결정성과의 화해책 (적용 A 제안, 적용 B 동의)

정규식 한계가 실전에서 바로 드러났다(적용 A: Fable5 "한도 개방" 왜곡을 못 잡아 케이스를 대체해야 했음. 적용 B: 결정론 토큰 대조는 원문에 있는 단어로 조립한 의미 왜곡을 원리적으로 못 잡는다). LLM 그레이더는 비결정이라 결정적 check와 충돌하는데, 해법은 **LLM 판정도 fixture처럼 동결하는 것** — freeze 때 판정을 저장하고 check는 그걸 replay, 루브릭 해시가 바뀔 때만 재판정. "LLM 판정의 결정화"가 도구의 다음 차별점이 될 수 있다.

### 7. 측정기가 프롬프트를 역으로 왜곡시킨 사례 — 문장 분리기 결함 (적용 B)

summary의 "3.0%"를 splitSentences류 분리기가 소수점에서 문장 경계로 오인해 "문장 3개" FAIL → 통과시키려고 프로덕션 프롬프트에 "소수점 수치는 원문 정수 표기 사용" 지시를 추가했다. **프로브 결함이 프로덕션 프롬프트 설계에 스며든 실사례**다(프로덕션 validate.mjs가 같은 분리기를 쓰므로 게이트 정합성은 있었지만, 근본은 분리기 결함). 프로브 작성 가이드에 소수점·약어("3.0%", "U.S.")를 문장 경계에서 제외하는 토크나이저 주의를 넣거나, 참조 구현을 제공할 것.

### 8. 재동결 마찰과 환경 불일치 — 한계 문서화 (적용 B)

- freeze/check --live는 로컬 claude CLI가 필수라 원격·모바일에서는 프롬프트를 못 고친다. CI workflow_dispatch로 re-freeze를 돌리는 레시피가 있으면 마찰이 준다.
- 계약 평가(단건 `claude -p`)와 프로덕션(클라우드 에이전트가 전체 스펙 맥락에서 일괄 생성)은 모델·컨텍스트·호출 방식이 다르다. 라이브 통과는 근사치이며 프로덕션 게이트를 대체하지 않는다 — README에 이 한계를 명시해야 "계약 통과 = 안전" 착시를 막는다.
- 결정적 check의 CI green이 뜻하는 것은 "계약을 아무도 안 건드렸다"이지 "프롬프트가 지금도 잘 작동한다"가 아니다(모델 드리프트·입력 분포 변화는 못 본다). 이것도 같은 맥락의 문서화 대상.

### 9. 표본 운영 수칙 (적용 B)

래칫은 표본만큼만 지킨다. 동결 6건이 하루치 뉴스라 어려운 유형(정치·사망사고·영문 고유명사 다수)이 비어 있다. "실전에서 어려운 입력을 만날 때마다 tests.yaml에 추가 → 라이브 통과 시 freeze로 편입"이 운영 수칙인데, 신규 케이스가 기존 동결과 어떤 관계로 편입되는지 문서화가 없다.

## 한 줄 요약

결정성·프로브 드리프트 검출은 진짜 우위니 전면에 세우고, 설치·등록 워크플로를 먼저 다듬고, 가장 큰 확장은 **라이브 lint 모드(프로브 이중화 해소)**와 **LLM 판정의 결정화**다. 그리고 두 실전 적용이 공통으로 증언하는 것: 도구의 적은 회귀만이 아니라 **프로브 자신의 결함과 이중화**다.
