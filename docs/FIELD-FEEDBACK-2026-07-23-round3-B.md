## P0 상세 — 사고 상태 그대로 재현해 검증

라운드 2에서 실제로 겪은 상태를 복원했다: 사본 ratchet.json에서 마지막 full freeze 스냅샷을 제거 → snap[0](6건, CCTV pass 포함) + snap[1](6건, CCTV 없음)의 부분 동결 상태.

- 결정적 `check` → **exit 0**, "floor 7건 전부 pass". v0.2.0에서 영구 반려되던 바로 그 상태에서 CCTV가 이전 스냅샷 출력으로 replay됐다 — 합집합 replay 작동 실증.
- probeHash 케이스별 분리도 관찰됨: snap[0]은 구버전 asserts로 동결된 상태라 `[probe drift]` 경고가 떴지만 하드 페일하지 않음(기본 정책 그대로).
- 라벨 분리: CCTV 동결 출력에 코드펜스를 인위 삽입 → `[회귀] …: 코드펜스 출력(금지)` — **일반 문구가 아닌 실제 assert 사유**로 표기 + `[drift]`(프로브 변경 영향 재채점 1→0.75)와 구분 표기, exit 1. 원복 후 exit 0.

**잔여 소견(사소)**: `[replay 불가]` 라벨은 replayMap 키에 없는 floor 케이스 전용인데, 정상 상태에선 합집합 replay 덕에 도달 불가한 방어 라벨이 됐다(좋은 일). 다만 **동결 케이스의 output 필드가 유실(손상)된 상태**를 시뮬레이션하면 케이스가 맵에 남아 eval을 타고 `[회귀] …: assert 실패`(일반 문구)로 찍힌다 — 라운드 2 오분류의 잔재 경로. replay-map 빌더가 output 없는 케이스를 커버 집합에서 제외하면 `[replay 불가]`로 정확히 분류될 것이다. 우선순위 낮음(손상 상태에서만 발생).

## P1 상세 — 재시도 경로를 운에 맡기지 않고 결정적으로 검증

1차 시도: 강화 전 프롬프트(라운드 2에서 8회 전패를 냈던 7dfb1a4 버전)로 `check --live --retry 2` → **초기 평가가 7/7로 통과해 재시도 미발동**(비결정성의 반대면 — 프로브가 모델 운에 의존하는 구조).

그래서 **래퍼 프로바이더**로 재시도 메커니즘을 결정적으로 검증했다: 케이스별 첫 호출은 고의 코드펜스 위반 출력, 재호출부터 실제 claude(케이스별 마커 파일로 구분).

- 결과: 7건 전부 초기 실패 → **재시도 발동** → "재시도 1/2: <케이스> → 통과" 진행 로그 + verdict에 케이스별 "**재시도 1/2에 통과**" 표기 → 최종 exit 0.
- **캐시 no-op 수정(d2789e6) 실작동 입증**: 재시도가 캐시된 위반 출력을 돌려받았다면 래퍼의 첫 분기 출력(코드펜스)이 그대로 와서 실패해야 하는데, 실제로는 래퍼의 두 번째 분기(실 claude)가 호출돼 통과했다. 케이스별 마커 파일 존재가 2회 실호출의 물증.
- 하위호환: `--retry` 미지정 시 기존 동작 동일(다수 실행으로 확인), 결정적 replay 경로에선 `[경고] --retry는 --live에서만 유효합니다 — 결정적 replay 경로에선 무시합니다` + 정상 진행 확인.

**제안(도구 개선 아님, 프로브 개선)**: CHANGES의 P1 재테스트 프로브는 "강화 전 프롬프트로 실패 재현"을 전제하는데, 모델 비결정성 때문에 재현이 보장되지 않는다(이번 실측: 어제 8회 전패했던 프롬프트가 오늘 1회에 7/7). 재시도 경로의 회귀 테스트는 위 래퍼 프로바이더 방식(고의 실패 주입)이 재현 가능하다 — 통합 테스트 픽스처로 편입할 가치가 있다.
