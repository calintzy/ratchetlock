## lint 경계 — 해소 (내 라운드2 항목 3 FAIL의 후속)

개발측이 내 3가지 제안 중 **3번(별개 도구로 정직히 문서화)** 을 채택하고, 1번(vars 어댑터)은 도구 내장이 아니라 **앱 측 패턴으로 문서화**(YAGNI 판단)했다. 이 결정이 라운드2 FAIL의 근본 우려를 정확히 덮는지 확인했다.

**내 FAIL 핵심 우려 = "vars 없는 lint가 위반을 놓치고 exit 0 → 통과로 오인"** — 아래로 커버됨:
- **README**: "`vars` 없이 부르면 그 프로브들은 전부 스킵되고, 위반이 있어도 exit 0으로 통과한 것처럼 보인다 — 위반을 놓친 것이지 없는 게 아니다."
- **AGENTS**: "exit 0을 곧이곧대로 '위반 없음'으로 보고하기 전에 [vars 필요 여부를 확인하라]" + 결정표에 "프로브가 vars를 쓰면 `--vars` 필수 — 없으면 스킵되어 놓친다" 명시. 에이전트의 착시 오보를 직접 금지.
- **PROBES**: "vars 어댑터 패턴 — lint는 항목+vars 단위다, 문서 전체가 아니다" + "계약의 수제 린터가 하던 일이 이 파싱·추정" + "계약 프로브(회귀 가드)와 라이브 표면 린터는 입력 모델이 다른 별개의 [도구]". 내 근본 원인 분석과 1:1 일치.

**이중 운영 판단을 문서가 지지하는가 — YES.** PROBES가 두 프로브를 "별개 도구"로 인정하고, vars 어댑터는 선택(앱이 필요하면 만드는 것)으로 둔다. 적용 A는 이미 수제 린터(`framing_lint.mjs`)가 그 파싱·추정을 하므로, 어댑터를 별도로 만드는 건 **파서 중복 = 과함**이다(CHANGES-round2가 "그게 과하면 이중 운영 유지가 정답"이라 명시). 따라서 현행 **계약(회귀 가드) + 수제 린터(라이브) 이중 운영 + BRIEFING_SPEC 정본·파생 동기화**를 유지한다.

**어댑터 풀 구현은 미수행(의도적)**: 수제 린터 파서를 lint 앞단에 복제하는 것이라 YAGNI. 대신 "vars를 주면 lint가 잡는다"는 이미 라운드2에서 실측 완료(`lint --output <위반> --vars <메타>` → exit 1). 어댑터의 나머지(md→vars 추정)는 수제 린터의 기존 로직과 동일하므로 새로 증명할 것이 없다.
