## 피드백 항목별 변경과 재테스트

### 1. 빌드·설치·영구 경로 (P0) — 부분 해결 + 근본 해결 준비

- **변경**: README 설치 섹션에 환경별 실측 분기 명시 — CI(ubuntu)는 `npm install --no-save
  github:calintzy/ratchetlock` 실측 동작, 로컬 macOS 일부 npm은 git 설치의 devDeps 미설치로 prepare가
  깨질 수 있어 클론+`npm link` 권장. `npm pack --dry-run`으로 배포 tarball(16파일, dist·replay-provider·
  LICENSE 포함) 검증 완료.
- **근본 해결**: npm 레지스트리 배포(패키지명 `ratchetlock` 선점 안 됨 확인). **배포는 npm 로그인 대기
  중** — 배포되면 모든 환경이 `npm install -g ratchetlock` 한 줄로 통일된다.
- **재테스트 (적용 A 환경)**: README 설치 절차(클론+link)를 영구 경로(~/ 아래)에서 그대로 실행 →
  `command -v ratchetlock && ratchetlock status` exit 0. 배포 완료 후에는 `npm install -g ratchetlock`
  → 같은 프로브. 실패했던 당시의 npm 버전(`npm -v`)을 보고에 남겨 주면 원인 좁히기에 쓴다.

### 2. 실전 등록 워크플로 (P0) — docs/REGISTER.md 신설

- **변경**: 등록 5단계(파일 4개 준비 → init → check --live → freeze 분기 → add-fail 승격) + CI 연동
  예시. demo.sh가 상태를 리셋한다는 경고 명시.
- **재테스트**: examples/cardnews를 **보지 않고** REGISTER.md만으로 신규 프롬프트 1건을 등록해 보라.
  프로브: 문서만 따라 해서 `ratchet.json` 생성 + freeze까지 도달하면 pass, 예제 역산이 필요했으면 fail
  (막힌 지점을 보고).

### 3. 라이브 lint (P1) — `lint` 커맨드 신설 + 단일 모듈 패턴 가이드

- **변경 ①**: `ratchetlock lint --output <파일> [--vars <JSON파일>] [--prompt <label>]` — 등록된 현재
  프로브를 임의의 새 출력 1건에 적용. LLM 호출 없음, ratchet.json 불변, 위반 시 exit 1.
  `.ratchet/last-eval.json`이 갱신되므로 잡힌 케이스를 `add-fail --from-last`로 곧장 승격 가능.
- **변경 ②**: docs/PROBES.md에 단일 모듈 패턴 — 프로덕션 검증기(validate.mjs류)와 계약 asserts가 같은
  CJS 모듈을 require해 이중화 드리프트를 없애는 구조와 예시 코드.
- **재테스트 (적용 A)**: 수제 린터를 `ratchetlock lint --output <오늘 브리핑 출력>`으로 대체 시도 →
  같은 위반을 잡으면(exit code 일치) pass. vars 의존 프로브는 `--vars`로 주입.
- **재테스트 (적용 B)**: PROBES.md 패턴대로 validate.mjs와 asserts.js가 공유 모듈 하나를 require하게
  리팩토링 가능한지 검토 → 가능하면 적용 후 `ratchetlock check` 여전히 exit 0 확인. 구조상 불가능한
  지점이 있으면 그게 다음 피드백이다.
