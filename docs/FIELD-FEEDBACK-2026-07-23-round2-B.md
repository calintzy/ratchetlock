## 발견 1. 부분 동결 × 결정적 check 구조 모순 — **P0**

`freeze --allow-partial`로 "이전 스냅샷에만 pass로 존재하는 케이스"가 생기면 **결정적 check가 구조적으로 항상 반려**한다. 실측 타임라인:

1. 프로브 9 수행 중 케이스 1건(CCTV)이 live 실패 → 정직하게 미동결(`--allow-partial`) → snapshot[신규]는 6건, CCTV는 이전 스냅샷에만 pass로 존재.
2. 결정적 `check` → exit 1, `[회귀] CCTV: assert 실패`. **CCTV의 동결 출력을 현재 프로브로 직접 재채점하면 pass인데도** 그렇다.
3. 원인(소스 확인, 별도 진단 레인 교차 검증): `deriveFloor`(floor.ts)는 activePrompt가 같은 **모든 스냅샷의 pass 케이스 합집합**을 floor로 요구(7건)하는데, replay는 `latestFrozenFor`가 주는 **최신 스냅샷 하나**만 replay-map에 적재(6건, check.ts:90). 커버 안 되는 케이스는 replay-provider.js:62-66이 `process.exit(1)` → 그 error 행이 `pass:false, failedAsserts:[]`로 정규화돼 findRegressions에서 **"assert 실패"로 오분류**(진짜 사유 "replay 불가"가 가려짐).
4. 우회는 전 케이스 동시 통과 후 **단일 스냅샷 full freeze**뿐 — 이번엔 그게 나와서 CI를 복구했지만, 안 나왔으면 계약이 결정적 check green 불가 상태에 갇힌다.

**왜 심각한가**: 이 시나리오는 오용이 아니라 README 표본 운영 수칙(어려운 입력을 부분 편입)이 권장하는 **정상 루프에서 자연 발생**한다. v0.1 플래그십(cardnews)의 OmniRoute는 어느 스냅샷에서도 pass인 적이 없어 floor 밖이라 안 드러났을 뿐이다.

**제안**: ①replay 소스를 최신 단일 스냅샷이 아니라 **floor를 덮는 스냅샷 합집합**으로(케이스별 최신 우선, probeHash 대조는 스냅샷별 분리) — 근본 해결. ②error 행을 "assert 실패"가 아닌 "replay 불가(스냅샷 커버 안 됨)" 별도 분류 — 최소 오진 방지. "floor에서 부분 동결 케이스 제외"는 래칫 철학 위반이라 지양.

## 발견 2. live 케이스별 재시도 정책 부재 — **P1** (실측 수치)

`check --live`/`freeze`는 형식 위반(JSON 앞 서두·코드펜스) 시 해당 케이스만 재생성하는 재시도가 없다. 케이스가 늘수록 전건 동시 통과가 확률 게임이 된다.

- 실측: 강화 전 프롬프트로 live **8회 연속** — 매회 7건 중 1~2건이 형식 위반, 동시 통과 0회. 실패는 무작위가 아니라 케이스별 재현성 있음(한 케이스는 동일 서두 문구 3연속, 다른 케이스는 코드펜스 4연속).
- 프롬프트로 해소된 과정(부산물 데이터 — 프롬프트 엔지니어링 관점에서 가치 있음): 일반 금지문("코드펜스 금지"·"메타 설명 금지")으로는 8회 전패. **실제 위반 문구를 negative example로 인용**("글자수 확인은 생략하고…"를 실패 예시로 명시)하고 "글자수 제약은 내부적으로만, 출력에서 언급 금지"를 추가하자 **직후 1회 만에 7/7**.
- 그래도 도구 제안은 유효: `--retry-format N`류 케이스별 재시도 옵션. 표본 확대(현재 19케이스 동결)가 권장 운영인 이상, 이 부재는 freeze를 점점 더 어렵게 만든다. 발견 1과 결합하면 악순환이다(부분 동결로 도피 → 결정적 check 사망).
