# 카드뉴스 재작성 프롬프트 — ratchetlock 플래그십 예제

이 예제는 실제로 운영하던 카드뉴스 재작성 프롬프트를 고치면서 겪은 일을 그대로 가져온 것이다.
과장된 시연용 프롬프트가 아니라, 진짜 v1과 진짜 v2가 있고, 둘 다 claude(sonnet)에 실제로 태워
받은 출력이 있다. ratchetlock이 이 v1 → v2 개선을 어떻게 "계약"으로 굳히는지 보여주는 게 목적이다.

## 무엇을 보여주는가

v1 프롬프트(`prompt.txt`)는 5개 테스트 케이스 중 **0/5**만 통과했다. 코드펜스를 쓰지 말라고 했는데도
4건이 코드펜스로 감싸 나왔고, 1건은 아예 JSON 파싱이 깨졌다. v2(`prompt_v2.txt`)로 다시 쓰고 나니
**4/5**까지 올라왔다. 나머지 1건(OmniRoute)은 코드펜스는 고쳐졌는데, 원문에 없던 "저자 주장" 한정어가
빠진 채로 나왔다 — 이건 로드맵에 미룬 문제가 아니라, 기존에 있던 프레이밍 체크가 v2 재작성에서
실제로 생긴 문제를 잡아낸 사례다. 그래서 헤드라인은 "0/5 → 4/5, 1건은 알고 있음"이다.

이 결과를 `demo.sh`가 재현한다. `init`으로 시작해서, v1이 진짜 0/5임을 확인하고, 그 실패들을
`add-fail`로 영구 등록하고, v2로 넘어가서 4건을 `freeze`하고, 마지막으로 결정적 `check`가 전부
통과하는 걸 exit 0으로 확인한다. 중간에 동결된 출력 하나를 일부러 망가뜨려서 `check`가 회귀를
잡아내는 것도 보여준다.

## 실행

```bash
bash examples/cardnews/demo.sh
```

기본값은 fixture 재생 모드다. claude 구독이나 API 키가 전혀 없어도 된다. `fixtures/`에 저장해 둔
실제 claude 출력을 그대로 되돌려주고, 현재 `asserts.js`로 다시 채점하는 방식이라서 LLM을 다시
부르지 않는다. 재실행해도 항상 같은 결과가 나오고, 끝나고 나면 `ratchet.json`도 원래 상태로
리셋된다 — 몇 번을 돌려도 안전하다.

실제 claude CLI로 처음부터 다시 평가해 보고 싶다면:

```bash
bash examples/cardnews/demo.sh --live
```

이건 claude CLI가 설치돼 있어야 하고(구독 계정, API 키는 불필요), 매번 새로 LLM을 호출하므로
결과가 fixture와 미세하게 달라질 수 있다.

## 각 단계에서 볼 것

1. **v1 실측** — `check --live`를 fixture로 돌리자마자 `.ratchet/last-eval.json`에서 뽑은 스코어카드가
   0/5를 보여준다. 이 시점엔 아직 아무것도 얼리지 않아서 `check` 자체의 판정(`floor 0건`)은
   트리비얼하게 통과한다 — 지킬 계약이 없으니 당연하다. 진짜 정보는 그 아래 스코어카드다.
2. **add-fail** — 코드펜스 3건과 JSON 파싱 실패 1건, 총 4건을 회귀 가드로 등록한다.
3. **B1: 프롬프트를 바꿔도 add-fail은 사라지지 않는다** — v2로 넘어가기도 전에, 아직 아무것도
   freeze하지 않은 상태에서 `check --live --prompt prompt_v2.txt`를 돌리면 `floor 4건`이 이미
   나타난다. add-fail로 등록한 계약은 어느 프롬프트로 갈아타든 계속 지켜야 한다는 뜻이다.
4. **freeze** — v2로 전환하며 4건을 동결한다. OmniRoute는 미동결로 남는다.
5. **결정적 check** — 이번엔 fixture를 다시 갈아 끼울 필요가 없다. 동결된 출력을 되돌리는 replay
   경로가 `check` 안에 이미 들어 있어서, 그냥 `check`만 돌리면 floor 4건이 전부 통과한다.
6. **회귀 시연** — 동결된 출력 하나에 코드펜스를 인위로 끼워 넣고 `check`를 다시 돌리면 즉시
   비정상으로 잡힌다. 원복하면 다시 통과한다.

실제 실행 로그 전문은 [TRANSCRIPT.md](TRANSCRIPT.md)에 그대로 남겨 뒀다.

## 프레이밍 체크와 "의미 왜곡 검출"은 다르다

`asserts.js`에는 저자 주장에 붙는 한정어("~라고 밝혔다", "~다고 합니다" 같은 표현)가 빠졌는지
보는 체크가 들어 있다. 이건 정규식 기반의 **결정적** 체크이지, LLM이 "이 문장이 원문 의미를
왜곡했는가"를 판단하는 루브릭이 아니다. 로드맵에 있는 "의미 왜곡 LLM 루브릭"은 이것보다 한 층
위의 문제이고, 이번 MVP 범위 밖이다. OmniRoute를 잡아낸 것도 이 결정적 체크이지, 의미를 이해해서
잡아낸 게 아니라는 걸 분명히 해 두고 싶었다.

## 왜 baseline.json과 baseline.regraded.json이 따로 있는가

`fixtures/baseline.json`은 파일럿 당시 구버전 프로브로 채점된 기록이고, `baseline.regraded.json`은
같은 출력을 지금 `asserts.js`로 다시 채점한 결과다. 둘이 다른 이유(프로브 자체를 교정한 경위)는
[fixtures/README.md](fixtures/README.md)에 자세히 적어 뒀다. 결론만 말하면, 프로브를 교정해도
v1의 0/5 헤드라인은 그대로다 — 딱 한 건(OmniRoute)의 세부 점수만 0.5에서 0.75로 올랐을 뿐, 코드펜스
때문에 여전히 fail이다.
